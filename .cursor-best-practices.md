# Cursor AI Best Practices for FPL DnD
## Safety, Security, Design & Deployability Guidelines

## 🔒 SECURITY BEST PRACTICES

### Environment Variables & Secrets
- **NEVER** commit API keys, tokens, or secrets to version control
- Always use environment variables (`.env.local`, `.env.production`)
- Use Next.js built-in environment variable handling
- Validate all environment variables at startup
- Use different secrets for development, staging, and production

### API Security
- Always validate and sanitize user inputs
- Implement rate limiting on API routes (`/api/*`)
- Use HTTPS for all external API calls
- Never expose sensitive endpoints without authentication
- Implement CORS properly for API routes
- Use Next.js API route middleware for authentication checks

### Session Management
- Use secure, HttpOnly cookies for session storage
- Implement session expiration and refresh logic
- Never store sensitive data in client-side storage (localStorage/sessionStorage)
- Use server-side session validation for protected routes
- Implement CSRF protection for state-changing operations

### Data Protection
- Sanitize all user inputs before database operations
- Use parameterized queries (if using database)
- Implement input validation on both client and server
- Never trust client-side validation alone
- Encrypt sensitive data at rest and in transit

### Authentication & Authorization
- Always verify user permissions server-side
- Use secure password hashing (bcrypt, argon2) if implementing auth
- Implement proper logout functionality
- Clear sessions on logout
- Use secure session tokens with expiration

## 🛡️ SAFETY BEST PRACTICES

### Error Handling
- Never expose stack traces or internal errors to users in production
- Implement proper error boundaries in React components
- Log errors server-side for debugging
- Provide user-friendly error messages
- Handle edge cases gracefully (null/undefined checks)

### Input Validation
- Validate all user inputs on the server side
- Sanitize data before rendering (prevent XSS)
- Use TypeScript for type safety
- Implement proper form validation
- Check for SQL injection risks (if using database)

### Resource Management
- Implement proper cleanup in useEffect hooks
- Cancel pending requests on component unmount
- Use AbortController for fetch requests
- Avoid memory leaks (remove event listeners, clear intervals)
- Implement proper loading states to prevent race conditions

### API Rate Limiting
- Implement client-side request throttling
- Use exponential backoff for retries
- Respect API rate limits from external services
- Cache responses when appropriate
- Monitor API usage

## 🎨 DESIGN BEST PRACTICES

### Responsive Design
- Always test on mobile, tablet, and desktop
- Use Tailwind responsive classes (`md:`, `lg:`, `xl:`)
- Ensure touch targets are at least 44x44px on mobile
- Test with different screen sizes and orientations
- Use relative units (rem, em) instead of fixed pixels where possible

### Accessibility
- Use semantic HTML elements
- Provide alt text for all images
- Ensure proper color contrast (WCAG AA minimum)
- Implement keyboard navigation
- Use ARIA labels where appropriate
- Test with screen readers

### Performance
- Optimize images (use Next.js Image component)
- Implement code splitting and lazy loading
- Minimize bundle size
- Use React.memo for expensive components
- Implement proper caching strategies
- Optimize API calls (batch requests, use React Query)

### User Experience
- Provide loading states for async operations
- Show error messages with retry options
- Implement optimistic UI updates where appropriate
- Use consistent spacing and typography
- Maintain consistent color scheme and branding
- Provide clear navigation and feedback

## 🚀 DEPLOYABILITY BEST PRACTICES

### Next.js Configuration
- Ensure `next.config.js` is properly configured
- Set proper environment variables for production
- Configure proper build settings
- Use static generation where possible
- Implement proper ISR (Incremental Static Regeneration) if needed

### Build Optimization
- Ensure build completes without errors
- Fix all TypeScript errors before deployment
- Remove console.logs in production (use proper logging)
- Optimize bundle size
- Test production build locally (`npm run build && npm start`)

### Environment Configuration
- Use different configs for dev/staging/production
- Never hardcode URLs or endpoints
- Use environment variables for all external services
- Document all required environment variables
- Provide `.env.example` file

### Database & Storage
- Use connection pooling if using database
- Implement proper migrations if using database
- Use environment-specific database URLs
- Implement proper backup strategies
- Use read replicas for production if needed

## 🌐 HETZNER CLOUD DEPLOYMENT

### Server Configuration
- Use Ubuntu 22.04 LTS or Debian 12 (stable, well-supported)
- Configure firewall (UFW) to allow only necessary ports:
  - 22 (SSH)
  - 80 (HTTP - for Let's Encrypt)
  - 443 (HTTPS)
- Disable root login, use SSH keys
- Keep system packages updated
- Use non-root user for application

### Domain & SSL Setup
- Configure DNS records (A/AAAA) pointing to Hetzner server IP
- Use Let's Encrypt for free SSL certificates
- Set up automatic certificate renewal (cron job)
- Configure Nginx or Caddy as reverse proxy
- Redirect HTTP to HTTPS
- Set proper security headers (HSTS, CSP, etc.)

### Application Deployment
- Use PM2 or systemd for process management
- Implement proper logging (rotate logs)
- Set up monitoring and health checks
- Configure automatic restarts on failure
- Use environment variables for configuration
- Implement zero-downtime deployments if possible

### Nginx Configuration (Example)
```nginx
server {
    listen 80;
    server_name yourdomain.com www.yourdomain.com;
    return 301 https://$server_name$request_uri;
}

server {
    listen 443 ssl http2;
    server_name yourdomain.com www.yourdomain.com;

    ssl_certificate /etc/letsencrypt/live/yourdomain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/yourdomain.com/privkey.pem;
    
    # Security headers
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-XSS-Protection "1; mode=block" always;

    location / {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
    }
}
```

## 📋 CODE QUALITY RULES

### TypeScript
- Always use proper types, avoid `any` when possible
- Fix all TypeScript errors before committing
- Use interfaces for object shapes
- Enable strict mode in tsconfig.json

### Code Organization
- Keep components small and focused
